## 1. 需求背景与目标

### 1.1 背景

个人用户在日常工作和生活中需要管理大量不同类型的任务（会议、待办事项、学习计划、生活杂务等）。传统任务管理工具（如纸质便签、基础 Todo 应用）存在以下痛点：

1. **创建成本高**：手动填写任务标题、截止时间、分类、优先级等字段，操作繁琐。
2. **优先级主观**：依赖用户自行判断紧急程度，缺乏客观分析依据。
3. **分类维护重**：需要手动维护分类体系，任务归类耗时且不一致。
4. **提醒被动**：仅支持固定时间提醒，无法结合用户习惯和任务上下文智能提醒。
5. **规划缺失**：任务列表是"堆砌式"的，缺乏对每日/每周时间容量的合理分配建议。
6. **千篇一律**：不学习用户行为偏好，所有用户获得完全相同的体验。

大语言模型（LLM）技术的成熟，使得上述痛点可通过 AI 能力有效解决。本项目旨在构建一个 B/S 架构的智能任务管理系统，将 AI 能力深度融入任务创建、排序、分类、提醒、规划、学习等全流程。

### 1.2 目标

引入 Smart Task Manager，实现以下目标：

1. **降低任务创建门槛**：用户通过自然语言一句话即可创建完整任务（含时间、地点、分类、优先级）。
2. **提供智能优先级排序**：基于任务紧急程度、截止时间、用户历史完成模式，自动计算并排序任务列表。
3. **实现零维护自动分类**：AI 根据任务语义自动归类，用户无需手动维护分类体系。
4. **构建智能提醒体系**：结合截止时间、任务重要性、用户活跃时段习惯，主动推送个性化提醒。
5. **提供时间规划建议**：AI 分析每日/每周任务容量，给出任务分布与执行顺序建议。
6. **实现个性化习惯学习**：系统持续学习用户行为偏好，越用越精准。

#### [REQ_TASK_MGMT_001] 任务 CRUD 基础操作

**需求描述**: 提供任务的完整生命周期管理，包括创建、查看、更新、删除、标记完成/未完成。这是整个系统的数据操作基础。

**详细规格**:

| 属性 | 说明 |
|------|------|
| 任务标题 | 必填，最大长度 200 字符 |
| 任务描述 | 选填，最大长度 5000 字符，支持纯文本 |
| 截止时间 | 选填，日期时间类型，支持无截止时间的开放式任务 |
| 优先级 | 必填，枚举值：`P0-紧急` / `P1-高` / `P2-中` / `P3-低`，默认值 `P2-中` |
| 分类 | 选填，关联分类表，初始分类集见 [REQ_TASK_MGMT_003] |
| 标签 | 选填，多选，关联标签表 |
| 状态 | 必填，枚举值：`pending` / `in_progress` / `completed` / `cancelled`，默认值 `pending` |
| 完成时间 | 系统自动记录，任务状态变更为 `completed` 时写入当前时间戳 |
| 创建时间 | 系统自动记录，任务创建时写入当前时间戳 |
| 更新时间 | 系统自动记录，任务任意字段变更时自动更新 |
| 位置/地点 | 选填，最大长度 200 字符 |
| 估计耗时 | 选填，单位为分钟，整数，范围 1-1440（最多 24 小时） |
| AI 生成标记 | 布尔值，标记该任务是否由 AI 智能创建/修改，用于习惯学习数据追溯 |

**操作规则**:
- 删除操作采用**软删除**（逻辑删除），将状态置为 `deleted`，不物理删除数据。
- 已完成任务可重新打开（状态从 `completed` 回退为 `pending`）。
- 任务列表支持按创建时间、截止时间、优先级、状态排序。
- 任务列表支持按分类、标签、状态、关键词（模糊匹配标题+描述）筛选。

**验收条件**:

| 编号 | 条件 | 优先级 |

| AC-001-02 | 创建任务后，任务立即出现在任务列表中，字段信息完整准确 | P0 |
| AC-001-03 | 编辑任务字段后保存，列表中对应任务信息同步更新 | P0 |
| AC-001-04 | 软删除任务后，该任务从默认列表视图中消失，但可通过回收站恢复 | P1 |
| AC-001-05 | 标记任务为完成后，完成时间字段自动填充当前时间戳 | P0 |
| AC-001-06 | 已完成任务可重新打开，完成时间字段清空 | P1 |
| AC-001-07 | 任务列表支持至少 4 种排序方式（创建时间、截止时间、优先级、状态） | P1 |
| AC-001-08 | 任务列表支持按分类、标签、状态、关键词组合筛选 | P1 |

#### [REQ_TASK_MGMT_002] 自然语言创建任务（AI_NLP）

**需求描述**: 用户在输入框中输入自然语言描述，系统调用 AI 模型解析出结构化任务信息（标题、截止时间、地点、分类、优先级等），自动填充任务创建表单。用户可在 AI 解析结果基础上修改确认后再正式创建。

**详细规格**:

**输入**: 自然语言文本字符串，最大长度 500 字符。

**AI 解析输出字段**:

| 输出字段 | 解析规则 | 默认值（无法解析时） |
|---------|---------|-------------------|
| 标题 | 从输入中提取核心任务内容 | 整条输入作为标题 |
| 截止时间 | 解析相对时间（"明天"、"下周三"）和绝对时间（"4月25日下午3点"），基于用户本地时区转换 | null（不设置截止时间） |
| 地点 | 提取地点实体（"在会议室A"、"去超市"） | null |
| 分类 | 语义判断归入哪个预设分类（见 [REQ_TASK_MGMT_003]） | null |
| 优先级 | 根据用词紧急程度推断（"赶紧"、"马上"→P0；"尽快"→P1；无特殊表述→P2） | P2-中 |
| 估计耗时 | 仅在用户明确提及（"大约1小时"）时提取 | null |

**交互流程**:
1. 用户在"智能创建"输入框中输入自然语言描述。
2. 用户点击"解析"按钮（或按 Enter 键）。
3. 系统调用后端 AI 解析接口，显示 loading 状态。
4. 返回结果后，在输入框下方展示**解析预览卡片**，以表单形式呈现解析出的各字段值。
5. 用户可修改任意字段值。
6. 用户确认无误后点击"创建"按钮，正式提交任务。

**异常处理**:
- AI 模型调用失败时，显示明确错误提示："AI 解析暂时不可用，请稍后重试或手动创建任务。"同时提供"手动创建"入口。
- AI 解析返回内容不合法（如标题为空）时，使用整条输入作为标题回退。
- 解析响应超时（>10 秒）时，中断请求并提示超时。

**验收条件**:

| 编号 | 条件 | 优先级 |
|-----|------|--------|
| AC-002-01 | 输入"下周三下午3点开会讨论方案"后，解析结果中标题为"开会讨论方案"，截止时间正确对应下周三 15:00 | P0 |
| AC-002-02 | 输入"明天去超市买牛奶"后，解析结果中地点为"超市" | P1 |
| AC-002-03 | 输入"赶紧提交报告"后，优先级自动设为 P0-紧急 | P1 |
| AC-002-04 | 解析预览卡片展示后，用户可修改任意字段并成功创建任务 | P0 |
| AC-002-05 | AI 模型调用失败时，显示友好错误提示并提供手动创建入口 | P0 |
| AC-002-06 | 解析响应超过 10 秒时，中断请求并提示用户 | P1 |
| AC-002-07 | 无截止时间信息的输入（如"整理桌面"），截止时间字段保持为空 | P0 |
| AC-002-08 | 不含任务语义的输入（如"你好"），给出提示"未能识别任务信息，请描述您要完成的事项" | P2 |

#### [REQ_TASK_MGMT_003] 自动分类与标签系统（AI_CLASSIFY）

**需求描述**: 系统预置一组常用分类，AI 根据任务内容自动归类。用户可自定义分类和标签。标签系统支持自动推荐和手动创建。

**详细规格**:

**预设分类（系统初始化时自动创建，用户可删除/修改/新增）**:

| 分类名称 | 分类图标 | 示例任务 |
|---------|---------|---------|
| 工作 | 💼 | "完成周报"、"准备会议材料" |
| 生活 | 🏠 | "去超市买菜"、"预约体检" |
| 学习 | 📚 | "阅读《深度学习》第三章"、"完成在线课程作业" |
| 健康 | 💪 | "跑步30分钟"、"预约牙医" |
| 社交 | 🤝 | "给朋友发生日祝福"、"组织同学聚会" |
| 财务 | 💰 | "缴纳水电费"、"检查信用卡账单" |

**自动分类规则**:
- 任务创建时（无论手动还是自然语言创建），后端调用 AI 分类接口对任务标题+描述进行语义分析。
- AI 返回最匹配的分类名称。如果匹配度低于阈值（70%），则不自动设置分类，留空让用户自行选择。
- 用户确认任务创建后，分类结果与任务关联保存。

**标签系统**:
- 每个任务可关联 0-N 个标签。
- 标签为纯文本，最大长度 30 字符，同一用户下标签名称唯一。
- AI 在任务创建时推荐最多 3 个相关标签，用户可选择采纳或忽略。
- 用户可随时手动添加/移除标签。
- 标签管理页面提供创建、编辑、删除、合并标签功能。

**AI 标签推荐逻辑**:
- 基于任务内容语义相似度 + 用户历史标签使用频率，综合推荐标签。

**验收条件**:

| 编号 | 条件 | 优先级 |
|-----|------|--------|
| AC-003-01 | 系统初始化后，预设的 6 个分类自动存在于分类列表中 | P0 |
| AC-003-02 | 创建任务"写周报"时，AI 自动归类到"工作"分类 | P0 |
| AC-003-03 | 创建语义不明确的任务（如"处理一下"），不强制设置分类 | P1 |
| AC-003-04 | 用户可新增自定义分类，新分类立即在分类选择列表中可用 | P1 |
| AC-003-05 | 用户可删除/修改预设分类和自定义分类 | P1 |
| AC-003-06 | 任务支持关联多个标签，标签可在任务编辑界面添加/移除 | P1 |
| AC-003-07 | AI 推荐标签功能正常，推荐结果与任务内容相关 | P2 |
| AC-003-08 | 标签管理页面支持创建、编辑、删除、合并操作 | P2 |

#### [REQ_TASK_MGMT_004] 智能优先级排序（AI_PRIORITY）

**需求描述**: AI 综合分析任务紧急程度、截止时间、重要性、用户历史完成模式等因素，计算每个任务的智能优先级分数，支持按此分数对任务列表排序。

**详细规格**:

**优先级分数计算因素（AI 综合评估）**:

| 因素 | 权重方向 | 数据来源 |
|------|---------|---------|
| 距截止时间 | 越近分越高 | 任务截止时间字段 |
| 用户设定优先级 | P0>P1>P2>P3 | 任务优先级字段 |
| 任务重要性 | AI 语义评估 | 任务标题+描述 |
| 历史完成率 | 该分类/标签下历史完成率高则降低紧迫感 | 用户行为数据 |
| 任务间依赖 | 被其他任务依赖的优先提升 | 任务依赖关系（可选，初版不实现） |
| 拖延风险 | 临近截止且未开始的任务大幅提分 | 任务状态 + 截止时间 |

**排序模式**:
- **AI 智能排序**：按 AI 计算的综合优先级分数降序排列（默认排序方式）。
- **手动排序**：用户可切换到手动拖拽排序模式。
- 用户选择排序方式后，偏好保存在前端 localStorage 中。

**触发时机**:
- 用户进入任务列表页面时自动计算一次。
- 任务新增/修改/状态变更后，重新计算受影响任务的分数。
- 每日凌晨自动全量重算一次（后端定时任务）。

**优先级分数展示**:
- 在任务列表中，可选展示"AI 推荐排序"标签/图标，让用户感知 AI 正在工作。
- 用户可悬停/点击查看 AI 给出该排序的建议理由（一句话）。

**验收条件**:

| 编号 | 条件 | 优先级 |
|-----|------|--------|
| AC-004-01 | 任务列表默认按 AI 智能优先级分数降序排列 | P0 |
| AC-004-02 | 距截止时间 1 天的未完成任务，排序位置高于距截止时间 7 天的任务 | P0 |
| AC-004-03 | 用户可在 AI 排序和手动排序之间切换 | P1 |
| AC-004-04 | 任务状态变更后，受影响任务的优先级分数自动更新 | P1 |
| AC-004-05 | 排序偏好（AI/手动）持久化到 localStorage，刷新页面后保持 | P2 |
| AC-004-06 | 每日凌晨后端定时任务自动全量重算优先级分数 | P1 |

#### [REQ_TASK_MGMT_005] 智能提醒（AI_REMIND）

**需求描述**: 系统根据任务截止时间、任务重要性、用户活跃时段习惯，智能决定提醒时机和频率，通过前端浏览器通知推送提醒。

**详细规格**:

**提醒策略层级**:

| 层级 | 触发条件 | 提醒内容 | 提醒方式 |
|------|---------|---------|---------|
| 截止日提醒 | 任务截止日当天 09:00（用户时区） | "📋 [任务标题] 今天到期" | 浏览器通知 |
| 提前一天提醒 | 任务截止日前一天 09:00 | "⚡ [任务标题] 明天到期，请做好准备" | 浏览器通知 |
| 紧急预警 | AI 判断该任务可能无法按时完成（拖延风险高） | "🔴 [任务标题] 存在逾期风险，建议优先处理" | 浏览器通知 + 页面内高亮 |
| 习惯时段提醒 | AI 学习用户活跃时段，在用户最活跃的时间窗口推送 | "💡 现在是高效时段，建议处理 [任务标题]" | 浏览器通知 |

**提醒规则**:
- 同一任务同类型提醒，24 小时内仅推送 1 次，避免重复打扰。
- 用户可对单个任务开启/关闭提醒。
- 用户可设置全局"免打扰"时段（如 22:00-08:00），该时段内不推送任何通知。
- 浏览器通知需要用户授权；未授权时，使用页面内通知条（in-page notification）替代。

**用户配置项**:
- 全局提醒开关：开/关
- 免打扰时段：开始时间、结束时间
- 单任务提醒开关：开/关（默认开）

**习惯学习数据来源（与 [REQ_TASK_MGMT_010] 关联）**:
- 记录用户每日活跃时间分布（浏览页面的时间段）。
- 记录用户完成任务的常见时段。
- 综合分析得出用户的"高效时段"窗口。

**验收条件**:

| 编号 | 条件 | 优先级 |
|-----|------|--------|
| AC-005-01 | 截止日当天 09:00，浏览器通知正确推送当天到期任务 | P0 |
| AC-005-02 | 截止日前一天 09:00，浏览器通知正确推送明天到期任务 | P0 |
| AC-005-03 | 同一任务同类型提醒 24 小时内仅推送 1 次 | P1 |
| AC-005-04 | 用户可开启/关闭单个任务的提醒 | P1 |
| AC-005-05 | 全局免打扰时段生效，该时段内不推送浏览器通知 | P1 |
| AC-005-06 | 浏览器通知未授权时，自动降级为页面内通知条 | P1 |
| AC-005-07 | AI 紧急预警对高拖延风险任务触发 | P2 |
| AC-005-08 | 习惯时段提醒在用户活跃高峰时段推送 | P2 |

#### [REQ_TASK_MGMT_006] 时间规划建议（AI_PLAN）

**需求描述**: AI 根据用户当前待办任务列表、每日可用时间、任务优先级和估计耗时，生成每日/每周任务执行规划建议。

**详细规格**:

**规划维度**:
- **日规划**：以"今天"为时间窗口，建议今天应该完成的任务及执行顺序。
- **周规划**：以"本周"为时间窗口，将本周到期任务合理分配到各天。

**输入参数**:
- 当前用户所有未完成任务列表（含优先级分数、截止时间、估计耗时）。
- 用户每日可用工作时间（默认值 8 小时，用户可自定义配置）。
- 历史完成速度（平均每天完成多少任务/多少小时的任务量）。

**输出内容**:

| 输出项 | 说明 |
|-------|------|
| 今日推荐任务列表 | 按建议执行顺序排列，标注理由（如"今天到期"、"优先级最高"） |
| 建议时间段分配 | 为每个推荐任务分配建议执行时间段（如"09:00-10:30 完成周报"） |
| 工作量评估 | 今日推荐任务总预估耗时 vs 可用时间，显示负载百分比 |
| 周视图 | 以甘特图/时间轴形式展示本周任务分布建议 |
| 过载预警 | 当某天推荐任务总量超过用户可用时间的 80% 时，红色高亮提示 |

**交互设计**:
- 页面提供"今日规划"和"本周规划"两个 Tab。
- 用户可点击"重新生成"按钮手动触发重新规划。
- 规划结果每天 08:00 自动刷新一次（后端计算，前端缓存）。
- 用户可拖拽调整规划中的任务顺序/时间段。

**验收条件**:

| 编号 | 条件 | 优先级 |
|-----|------|--------|
| AC-006-01 | 今日规划页面展示推荐的今日任务列表，按建议顺序排列 | P0 |

#### [REQ_TASK_MGMT_010] 基于习惯学习（AI_LEARN）

**需求描述**: 系统持续收集和分析用户行为数据，学习用户偏好模式，将学习成果应用到智能优先级排序、智能提醒、时间规划等功能中，实现"越用越懂你"。

**详细规格**:

**学习数据维度**:

| 数据维度 | 采集方式 | 应用场景 |
|---------|---------|---------|
| 活跃时段 | 记录用户每日浏览页面的时间段分布 | 智能提醒推送时机 |
| 完成时段 | 记录任务完成的实际时间段 | 时间规划建议 |
| 分类偏好 | 统计各分类下任务创建和完成频率 | 自动分类优先级 |
| 标签偏好 | 统计各标签的使用频率 | 标签推荐排序 |
| 完成速度 | 统计不同分类/优先级任务的平均完成耗时 | 时间规划中的耗时预估 |
| 推理接受率 | 统计 AI 建议（排序/分类/标签）的用户采纳率 | AI 推理权重调整 |
| 拖延模式 | 统计任务在截止时间前的完成分布（提前完成 vs 卡点完成 vs 逾期完成） | 拖延风险预测 |

**学习机制**:
- 后端维护一张 `user_behavior_log` 表，记录每次关键行为（任务创建、完成、修改、AI 建议采纳/拒绝）。
- 后端每日凌晨运行学习分析任务，将原始行为数据聚合为偏好模型参数，存入 `user_preference` 表。
- AI 各功能调用时，查询 `user_preference` 表获取用户偏好参数，融入推理 Prompt。
- 学习周期：连续使用 7 天后开始产出有效偏好数据（不足 7 天时使用系统默认值）。

**用户控制**:
- 提供"偏好学习"设置页面，展示系统已学习到的偏好摘要（如"您的高效时段是 09:00-11:00"）。
- 用户可手动修正偏好（如手动设置"我的高效时段是 14:00-16:00"），修正值优先于学习值。
- 用户可一键清除所有学习数据（"重置学习数据"按钮），系统回退到默认行为。

**隐私原则**:
- 所有行为数据仅存储在用户自有数据库中，不上传至第三方。
- AI 模型 API 调用时，仅发送必要上下文，不发送原始行为日志明细。

**验收条件**:

| 编号 | 条件 | 优先级 |
|-----|------|--------|
| AC-010-01 | 每次关键行为（创建/完成/修改/采纳AI建议）正确写入行为日志 | P1 |
| AC-010-02 | 每日凌晨学习分析任务正常执行，偏好参数更新到 user_preference 表 | P1 |
| AC-010-03 | 持续使用 7 天后，偏好学习数据开始影响 AI 推理结果 | P2 |
| AC-010-04 | 偏好学习设置页面展示已学习到的偏好摘要 | P2 |
| AC-010-05 | 用户可手动修正偏好参数，修正值优先于学习值 | P2 |
| AC-010-06 | 用户可一键清除所有学习数据，系统回退到默认行为 | P2 |
| AC-010-07 | 使用不足 7 天时，所有功能使用系统默认参数，不报错 | P1 |

#### [REQ_TASK_MGMT_012] AI 模型配置

**需求描述**: 系统通过环境变量配置 AI 模型连接参数，不绑定特定模型厂商，用户可灵活切换。

**新增配置项**:

| 配置项 | 环境变量 | 类型 | 默认值 | 说明 |

### 3.1 性能指标

| 指标 | 目标值 | 测量方法 |
|-----|--------|---------|
| 任务列表页面首屏加载时间 | ≤ 2 秒（100 条任务以内） | Chrome DevTools Performance 面板 |
| AI 自然语言解析响应时间 | ≤ 5 秒（95 百分位） | 后端日志统计 API 响应时间 |
| AI 优先级排序计算时间 | ≤ 3 秒（500 条任务以内） | 后端日志统计 |
| 任务 CRUD 接口响应时间 | ≤ 200 毫秒（95 百分位） | 后端日志统计 API 响应时间 |
| AI 时间规划生成时间 | ≤ 8 秒（100 条未完成任务以内） | 后端日志统计 |

### 3.2 可靠性指标

| 指标 | 目标值 | 说明 |
|-----|--------|------|
| AI 功能降级 | AI 模型不可用时，基础功能（CRUD、手动排序、手动分类）不受影响 | 系统应优雅降级，非功能阻塞 |
| 数据持久化 | 任务数据持久化到 MySQL，无内存丢失风险 | 所有数据写入数据库 |
| 数据库自动备份 | 每日凌晨自动备份数据库 | Docker 部署时通过 cron job 或 volume snapshot 实现 |
| 错误恢复 | AI 调用失败自动重试（最多 2 次），重试仍失败则返回明确错误 | 配置见 [REQ_TASK_MGMT_012] |

### 3.3 安全性需求

| 需求 | 说明 |
|-----|------|
| 密码加密存储 | 用户密码使用 bcrypt 哈希存储，禁止明文存储 |
| JWT 安全 | JWT Token 使用 HS256 算法签名，密钥通过环境变量 `JWT_SECRET` 配置 |
| SQL 注入防护 | 使用 ORM（SQLAlchemy）参数化查询，禁止拼接 SQL 字符串 |
| XSS 防护 | 前端 React 默认转义输出，禁止使用 `dangerouslySetInnerHTML` |
| CORS 配置 | 生产环境仅允许同源访问，开发环境允许 localhost |
| API Key 保护 | AI API Key 仅存储在后端 .env 中，禁止暴露到前端代码或 API 响应中 |
| 认证保护 | 所有 API 接口（除注册/登录/健康检查外）必须携带有效 JWT Token |

## 8. 风险与缓解

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| AI 模型 API 响应延迟高 | 自然语言解析、智能排序等功能体验差 | 中 | 设置合理超时（10 秒），前端展示 loading 状态和进度提示；AI 调用失败时优雅降级到手动操作 |
| AI 模型解析准确率不足 | 自然语言创建任务时解析结果需要大量手动修正，降低产品价值 | 中 | 设计"解析预览+确认"交互，用户可快速修正；持续优化 AI Prompt；提供"手动创建"作为回退方案 |
| 国产 AI 模型 API 不稳定 | 频繁超时或返回错误，影响 AI 功能可用性 | 中 | 实现自动重试机制（最多 2 次）；支持快速切换不同模型厂商（仅改 .env）；AI 不可用时基础功能不受影响 |
| 习惯学习冷启动问题 | 新用户使用初期（<7 天）无法提供个性化体验 | 高 | 明确告知用户需要使用一段时间才能获得个性化建议；冷启动期间使用系统默认参数 |
| 单用户架构限制 | 未来如需支持多人使用，需大规模重构 | 低 | 本期明确为单用户，数据模型预留 `user_id` 字段便于未来扩展 |
| 浏览器通知权限问题 | 用户拒绝授权浏览器通知，无法接收智能提醒 | 中 | 未授权时自动降级为页面内通知条；提供设置引导用户开启通知权限 |
